## 비교 기준

시장가격형:
- 기준일 현재값
- 전주 대비 변화율
- 4주 누적 변화율
- 향후 최근 3년 주간 변화 백분위
- 필요한 경우 최근 흐름 또는 추세 판정
- 휴일과 결측일에는 목표일 이하의 가장 최근 유효 관측값 사용
- 실제 비교에 사용한 관측일 보존

정기발표형:
- 이전 발표치
- 시장 예상치
- 3개월 추세

근원 PCE:
- 월간 발표형 지표로 처리한다.
- FRED 지수 수준에서 전월비 변화율을 계산한다.
- 이전 발표치와 3개월 전월비 평균을 기본 비교 기준으로 사용한다.
- 시장 예상치는 신뢰할 수 있는 출처가 확보되기 전까지 `null`로 둔다.

## 공통 출력

- 모든 지표는 공통 필드와 유형별 `metrics`를 가진 정규화된 객체로 출력한다.
- 공통 필드는 `indicatorId`, `name`, `type`, `asOf`, `unit`, `available`, `status`, `source`, `metrics`, `error`를 포함한다.
- `status`는 데이터 수집 단계에서 판정하지 않은 경우 `null`을 허용한다.
- 계산은 원래 정밀도를 유지하며 반올림은 보고서 표시 단계에서만 수행한다.
- 상세 계약은 `data/schema/indicator-output.schema.json`을 따른다.

## 결측 및 실패 처리

- 지표 1개가 실패해도 나머지 지표 수집과 보고서 생성은 계속한다.
- 실패한 지표는 `available: false`로 표시한다.
- 실패한 지표의 `metrics`는 `null` 또는 스키마에 맞는 빈 구조를 사용한다.
- `error`에는 민감하지 않은 코드와 요약 메시지만 기록한다.
- 지표 누락 시 최종 보고서 신뢰도를 낮춘다.
- 핵심 지표 2개 이상 실패 시 보고서 생성을 중단한다.
- 핵심 지표의 범위는 별도 결정 전까지 미해결 상태로 둔다.

## 데이터 보관

- 운영 원시 시계열은 공개 저장소에 장기간 보관하거나 커밋하지 않는다.
- 현재값과 변화율이 포함된 개인 최종 보고서는 공개 저장소에 커밋하지 않는다.
- 공개 저장소에는 코드, 설정, 임계값, 문서, 데이터 스키마, 사람이 만든 합성 데이터, 비민감 예시 파일을 보관할 수 있다.
- Notion 연동 전에는 최종 보고서를 수동 실행 화면에만 출력한다.
- Notion 연동 후에는 최종 보고서를 개인 Notion 데이터베이스로 전송한다.
- Telegram 연동 후에는 핵심 요약과 경고만 전송한다.

## Telegram 주간 알림

목적:

- Notion을 전체 주간 보고서의 원본으로 사용하고 Telegram은 핵심 요약과 경고를 빠르게 확인하는 채널로 사용한다.
- 전체 Markdown 보고서나 weekly-report-output 전체 JSON을 Telegram에 복사하지 않는다.

정상 주간 알림:

- 정상 위험 단계도 포함해 주간 보고서가 성공할 때마다 한 메시지를 전송한다.
- 기준일, 전체 위험 단계, 전체 위험 점수, 신뢰도, 핵심 변화 최대 3개, 취약 테마 최대 3개, 권장 대응을 포함한다.
- 전체 위험 단계가 `alert` 또는 `high_risk`이면 같은 메시지에 경고 머리말을 사용하며 별도 중복 경고 메시지는 보내지 않는다.
- 신뢰도가 `reduced`이면 데이터 누락이나 상충 신호로 판단 신뢰도가 낮아졌다는 고정 경고 문구를 포함한다.
- 전체 보고서는 Notion에서 확인하도록 안내하되 MVP에서는 Notion page 직접 링크를 포함하지 않는다.
- 취약도는 기대수익률이나 직접적인 매도 신호가 아니라 매크로 노출도라는 주의 문구를 포함한다.

데이터 품질 실패 알림:

- `quality.shouldAbort === true`이면 정상 주간 보고서와 Notion 저장을 진행하지 않고 데이터 품질 실패 알림을 한 메시지로 전송한다.
- 기준일, 보고서 생성 중단 안내, 실패한 핵심 지표 ID 최대 5개, 안전한 warning code 최대 3개만 포함한다.
- 계산되지 않은 전체 위험 단계·점수·권장 대응을 추정하거나 생성하지 않는다.

형식과 길이:

- `HTML` parse mode를 사용하고 모든 동적 문자열을 escape한다.
- 보이는 텍스트 기준 최대 3,500자로 제한해 Telegram의 4,096자 제한 아래 여유를 둔다.
- MVP에서는 메시지를 분할하지 않고, 항목 수와 각 항목 길이를 제한하며 선택 세부내용부터 축약해 한 메시지로 유지한다.
- 제한된 입력으로도 3,500자를 초과하면 전송하지 않고 `TELEGRAM_SUMMARY_TOO_LONG`으로 실패한다.

중복과 보안:

- MVP에서는 동일 기준일 workflow 재실행에 따른 Telegram 중복 전송을 허용한다.
- 정상 알림의 논리적 delivery key는 `telegram-weekly:{asOf}`, 품질 실패 알림은 `telegram-quality:{asOf}`로 정의하지만 저장하거나 exactly-once 보장에 사용하지 않는다.
- 실제 개인 보유 수량, 평가금액, 계좌별 비중을 전송하지 않는다.
- API token, chat ID, 메시지 본문, Telegram 원문 응답을 로그에 남기지 않는다.

## 로그

- GitHub Actions 로그에 원시 시계열 전체를 출력하지 않는다.
- 오류 로그에 API 응답 본문 전체를 출력하지 않는다.

## 보고서

- 5분 이내에 핵심 판단 가능
- 전체 위험 단계
- 영역별 위험
- 지표 코멘트는 특이사항이 있을 때만
- 포트폴리오 취약도는 테마 단위
- 기본적으로 상위 3개 취약 테마 표시
- 헷지는 필요성이 있을 때만 1~2개 제안
- 다음 주 조건과 판단 기록
- 세부 출력 규칙은 `docs/REPORT_SPEC.md`를 따른다

## 특이사항 조건

- 과거 분포상 이례적 움직임
- 전쟁·관세·정책·공급 충격
- 다른 지표와 상충
- 예상치와 유의미한 차이
- 추세 전환 또는 주요 기준선 돌파
- 데이터 오류·휴장·발표 지연

## 자동화 범위

AI 없이:
- 데이터 수집
- 변화율
- 임계값
- 영역 점수
- 테마 취약도
- 표 생성
- Notion 저장

AI:
- 사건 요약
- 상충 신호 설명
- 일시적/지속적 변화 구분
- 포트폴리오 영향
- 헷지 설명
- 자연어 보고서
